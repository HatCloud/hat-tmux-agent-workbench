## 执行摘要

1. **【Critical · 已实证】tmux.conf 的 `run-shell` 未对 `#{window_name}` 做 shell-quote，构成可达的 RCE。** 项目在**所有 hook**（tmux.conf:200-212）都正确用了 `#{q:...}`，唯独 popup/prompt 这批绑定（74、87-89、164、166）用裸 `'#{window_name}'`——而 `window_name` 恰恰是这些 format 变量里**唯一携带不可信内容**的（codex 用整段 prompt 当 session title，`claude_session.go:907`）。实证：窗口名设为 `x'; touch /tmp/hat_audit_pwned; echo '` 后，`tmux run-shell -t <win> "echo '#{window_name}'"` 真的创建了该文件。**触发只需：粘一段含单引号的不可信文本给 codex → 按 `alt-s`/`prefix w`/`prefix T` 任一日常键。**

2. **【根因洞察】三个最严重问题同源：codex 拿整段 prompt 当窗口名。** 这份**不可信 + 无界**的数据流进了 tmux 的多条路径，各自造成不同后果：内存泄漏（已修）、shell 注入（未修）、window option 无界写入。本轮的 `truncateWindowTitle` 修复只处理了"无界"，**没有处理"不可信"**——100 字符足够塞下 `'; rm -rf ~ ; '`。

3. **【Important · 已实证】daemon 在持全局锁期间做无超时外部调用，tmux 变慢即冻结整个 daemon。** `finishTask`（`tracker-server/main.go:455` 加锁）在 `:492` 持锁调 `isWindowWatched()`，其调用链上的 `exec.Command`（`:1199,1283,1287,1301`）**全部无 context/超时**；而同仓库 `cmd/agent/main.go:876-895` 的对应函数**都有 3s 超时**。常驻且持锁的 daemon 反而比一次性 CLI 防护更弱。与内存 bug 互相放大：6.8GB 被 swap 的 tmux 响应变慢，恰是触发条件。

4. **【Important】一致性完全靠文档里的人肉约定维持，无任何自动化闸门。** 无 golangci-lint / staticcheck / jscpd / shellcheck（shellcheck 已装但未接入），shell 侧零测试，Go 覆盖率 10-15%。CLAUDE.md 里"改动时要保持脚本与 Go 一致"这类提醒有多处，全靠人读到。**今天的 6.8GB bug 能活到今天，是同一个成因**——ARCHITECTURE.md 甚至白纸黑字写着"数据层不截断"这条导致 bug 的设计。

5. **【Important】可观测的死代码规模不小**：Devices 面板残留（CLAUDE.md 自称已剔除）+ 4 个死枚举 + 9 个零调用函数 + 6 个孤儿脚本，均已用 grep 命中数=1 核实。

## Findings

### Critical

#### C-1 `[shell/安全] tmux.conf 的 run-shell 未 shell-quote `#{window_name}`，构成可达 RCE` — `tmux/tmux.conf:74,87-89,164,166`

- **证据**：
  - 漏防护点：`tmux.conf:88` `bind T run-shell "...open_agent_palette.sh '#{client_tty}' ... '#{window_name}' 'todos'"`；同形态见 `:164`（`M-s`）、`:166`（`prefix o`）、`:74`（`command-prompt` + `%%`）、`:87`（`prefix t`）、`:89`（`prefix w`）。
  - 对照组：`tmux.conf:200-212` 的**全部** hook 均用 `#{q:session_id}`/`#{q:window_id}`/`#{q:pane_id}`——项目知道正确写法。
  - 污染源：`claude_session.go:907` `liveTitle = codexMeta.Title`（codex 用整段 prompt 当 title）；`docs/ARCHITECTURE.md:51` 亦载明。
  - 清洗缺口：`sanitizeWindowMarker`（`claude_session.go:1710-1719`）只剥 C0/C1 与 `#`，**不剥单引号**；且按 `:961,986` 只作用于 `@agent_title`，不作用于 codex 的 `liveTitle`。
  - **实证**（本次审计实跑，无害 payload，已清理）：
    ```
    tmux rename-window -t $W "x'; touch /tmp/hat_audit_pwned; echo '"
    tmux run-shell -t $W "echo 'PAYLOAD_BOUNDARY:#{window_name}'"
    → /tmp/hat_audit_pwned 被创建
    ```
- **影响**：把一段含单引号的不可信文本（网页/issue/代码片段）粘给 codex 当 prompt，窗口即被自动改名为该原文；此后用户按 `alt-s`/`prefix o`/`prefix w`/`prefix T`/`prefix t` 任一**日常高频键**，即触发任意命令执行。全程无需用户意识到"标题含特殊字符"。这是 prompt injection → shell RCE 的完整链路。
- **修复方向**：所有 `run-shell` 里的 format 引用改 `#{q:...}`；`%%`（command-prompt 用户输入）在 format 引擎展开前就已替换、无 `#{q:}` 等价物，须改用 argv 传递或临时文件，或对含 `'` 的输入做拒绝。
- **注意**：本轮已合入的 `truncateWindowTitle` **不修复此项**——它只截长度、不剥引号。

### Important

#### I-1 `[性能/健壮性] finishTask 持全局锁做无超时外部调用，tmux 变慢即冻结整个 daemon` — `cmd/tracker-server/main.go:449-497`

- **证据**：`:455` `s.mu.Lock()` + `defer s.mu.Unlock()`；`:492` `t.Acknowledged = isWindowWatched(target.WindowID)` 在锁内。调用链 `isWindowWatched`→`isActiveWindow`/`terminalIsFrontmost`→`frontmostBundleID`（`:1264-1298`，调 `lsappinfo`）。daemon 侧 exec 全部无 context：`:1199,1283,1287,1301` 均为裸 `exec.Command`。对照 `cmd/agent/main.go:876-895` 的 `runTmux`/`runTmuxOutput` **都用 `exec.CommandContext` + 3s 超时**。
- **影响**：`finish_task` 在每次 busy→idle 宽限满 2s 后都会调用（常规路径，非边缘）。若此刻 tmux 响应慢（**如本轮那个 6.8GB 被 swap 的 tmux server**）或 `lsappinfo` 挂起，daemon 在持进程级单锁状态下卡死——所有窗口的 `start_task`/`finish_task`/`acknowledge`/`mark_asking` 全部阻塞，直到调用返回或进程重启。与内存 bug 互相放大。
- **修复方向**：daemon 侧 exec 统一套 3s 超时（对齐 `cmd/agent` 模式）；`isWindowWatched` 探测移出临界区（先探测再加锁写状态）。

#### I-2 `[shell] new_session.sh 的锁无 trap 与陈旧恢复，进程被杀会永久关闭新 session 自动编号` — `tmux/scripts/new_session.sh:8,34`

- **证据**：`:3` `LOCK="/tmp/tmux-new-session.lock"`，`:8` `touch "$LOCK"`，`:34` `rm -f "$LOCK"`，**中间无 trap**。`session_created.sh:4-6` 用 `[ -f "$LOCK" ]` 直接短路跳过自动编号。对照 `reflow_agent_layout.sh:20-31`（mkdir 锁 + 10s 陈旧覆盖）与 `tracker_cache.sh`（`trap ... EXIT`）——项目其它锁都有兜底，唯独这里两者皆无。
- **影响**：`new_session.sh` 在 touch 与 rm 之间被杀（pkill / 睡眠打断 / `session_manager.py` 挂起），锁永久残留，此后**所有**新建 session 静默跳过自动编号，直到手动 `rm /tmp/tmux-new-session.lock`。
- **修复方向**：加 `trap 'rm -f "$LOCK"' EXIT`，或比照 `reflow_agent_layout.sh` 加 mtime 陈旧覆盖。

#### I-3 `[性能] window_task_icon.sh 每次状态栏重绘、每窗口重新 fork cat+jq 全量解析 tracker cache` — `tmux/tmux.conf:185,187`

- **证据**：`tmux.conf:185` `window-status-format` 内联 `#(...window_task_icon.sh "#{window_id}" ...)`，**per-window 求值**；`window_task_icon.sh:23,27-35` 每次调用先 `tmux list-panes`、再 `cat "$CACHE_FILE"` 整份文件交给 `jq` 过滤单个 window_id。同构问题见 `session_task_icon.sh:9-16`。`tmux.conf:172` `status-interval 3`。
- **影响**：N 个窗口每次重绘 = N × (fork tmux + fork cat + fork jq + 重新解析整份 JSON)；窗口切换/resize 也触发。cache 随任务数增长，属"每 tick 对同一份数据做的工作量随规模超线性增长"——与已修 bug 同属「热路径吞进不该吞的数据量」，只是数据源是外部文件而非 tmux 内部 format。
- **修复方向**：icon 判定收进 Go daemon 一次算好（参考 `right.sh` 直接 `exec agent tmux right-status`），或至少让 jq 一次性产出 window_id→icon 映射表、渲染时只查表。

#### I-4 `[性能] agentWindowName 每窗口每 tick 发起 30+ 次独立 tmux 子进程` — `cmd/agent/claude_session.go:804-1048`

- **证据**：函数体内 `tmuxWindowOption`/`runTmux("set"...)` 调用点共 33 处，每个 `tmuxWindowOption` 都是独立 `show-options` 子进程（`main.go:738-748`）。加上同轮的 `agentAIPane`/`reconcileSSHPaneBorder`/`reconcileSSHHost`/`reconcileWindowOrientation`，单窗口单 tick 保守 40-50 次 `exec.Command("tmux", ...)`。
- **影响**：fork 数 ≈ 窗口数 × 50 / interval（poll 可配至 500ms）。非无界内存增长，是本可用一次 `list-windows -F "#{@opt1}::#{@opt2}..."` 批量完成的事被拆成几十次进程调用。
- **修复方向**：批量化窗口选项读取。

#### I-5 `[重复] 状态前缀词表 [B]/[I]/[?]/[L]/[E] 四处硬编码，跨两个 binary` — `claude_session.go:661-679,1056-1063,1065-1073` + `tracker-server/remote_bell.go:75-89`

- **证据**：`statusTag()`（权威 switch）、`extractStatusPrefix()`（slice 重列）、`stripStatusPrefix()`（链式 TrimPrefix 三列）、`remoteWindowStatusPrefix()`（**另一 binary**，反向映射第四列，注释自陈"mirrors exactly what the user would see"）。
- **DRY 自检**：**knowledge duplication**——同一份业务词表，新增 `[W]` 或改前缀格式要同时改 4 处跨 2 package，编译器不报错。`internal/tracker/tracker.go:5-11` 已有 `Status` 类型但未覆盖这层词表，抽取点现成。
- **Clone Type**：Type-2。**修复方向**：建单一 `[]struct{Tag, Status string}` 表，挪进双方可 import 的 `internal/statustag`。

#### I-6 `[耦合] @agent_pane_role 跨 Go/shell 无共享定义，typo 静默失效` — `build_agent_layout.sh:38,56,58` ↔ `claude_session.go:746,1359,1456`

- **证据**：shell 写字面量 `ai`/`git`/`run`；Go 读 `#{@agent_pane_role}` 后与字面量比较。
- **Connascence**：Meaning · **跨进程** · Degree = 3 值 × 4 读点 × 3 写点，无编译期校验。
- **影响**：拼写不一致时 `layoutProportionsOK`/`reconcileWindowOrientation` **不报错**，只是判断失真或跳过 reflow——用户看到布局比例长期错误却无任何日志。失败模式静默是定 Important 的关键。
- **修复方向**：Go 侧 `const paneRoleAI = "ai"` 集中定义；找不到匹配角色时输出可观测日志，把静默失效改成可追踪。

#### I-7 `[耦合/重复] 朝向阈值跨进程 Algorithm connascence，靠 CLAUDE.md 人肉提醒维持` — `orientation_for_window.sh:21` ↔ `claude_session.go:1498-1501`

- **证据**：shell `w*10 >= h*20`（建窗硬判 2.0）；Go `w*10>=h*22` / `w*10<=h*18`（运行时滞回带）。
- **核实结论**：**数值不同是设计如此，不是漂移、不是 bug**（两侧注释各自说明用途，与 CLAUDE.md 描述吻合）。
- **Connascence**：Algorithm · 跨进程 · Degree=2。定 Important 是**按结构风险**：这份"保持一致"的约定完全靠 CLAUDE.md 人肉提醒，无共享常量、无跨语言断言。改一侧而没读到文档，就真的劣化为漂移。
- **修复方向**：**不要统一数值**。可选：shell 改调 Go CLI（`agent tmux orientation-for`），从"两份实现 + 文档提醒"降为"一份实现两处调用"。或至少把共享的物理假设（终端字符高≈宽 2 倍）提成具名锚点。

#### I-8 `[重复] Go↔shell 双写：worktree 主仓库解析 与 路径缩写` — `main.go:626-654`/`tmux/scripts/agent:120-137`；`main.go:572-594`/`tmux/scripts/agent:97-113`

## 附：用户追加需求（内存监控 + 回收）

**「定期自动回收 tmux 内存」技术上不可行**，规划时务必据此调整：tmux 无 GC / compact / trim API，malloc 过的内存不还给系统；唯一回收手段是重启 tmux server = 杀掉全部 session。自动定期重启不可接受。

可行的替代组合：

1. **内存监控**：daemon 定期采样 tmux server 内存，超阈值经已有 `terminal-notifier` 链告警（daemon 已在用，见 `notifyResponded`/`notifyAsking`）。
   - **取值坑（重要）**：不能用 `ps -o rss=`——实测 RSS 687MB vs 真实 footprint 6.8GB，macOS 内存压缩使 RSS 严重低估。Activity Monitor 的 "Memory" 列 = `phys_footprint`。命令行 `footprint -p <pid>` 可取但每次 fork。Go 侧轻量取法待 spike：`proc_pid_rusage()`（libproc）的 `ri_phys_footprint`，需 cgo 或 unsafe syscall。
   - 阈值建议可配（`agent-config.json`），缺省 ~2GB。
2. **「回收」= 引导到已有机制**：告警时提示走 `prefix s` 存档 → 重启 tmux → `tmux-resume` 恢复（见 `docs/GUIDE_WORKSPACE.md`）。把不可自动化的那一步交给用户按，而非背着用户重启。
